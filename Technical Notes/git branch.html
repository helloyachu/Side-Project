  • branch

1.在github上用pull request

進入github new-feature branch，點選Compare & pull request(綠色按鈕)或是Contribute項下的Open pull requesst(綠色按鈕)

Can’t automatically merge.，是因為在A電腦跟B電腦都做過更新了，這兩個branch是有衝突的狀態，有衝突的狀態下無法直接merge

要使用rebase，在主分支master有做修改的情況下，將new-feature接回原本的分支的最新進度。

(B電腦)
指令:git switch master
(目前修改中的檔案要先commit才能切換)

switch補充
如果開發還沒有完成，要做switch，可以先做假的commit，commit訊息輸入’work in Progress’，就可以切換到別的分支，工作的時後在切回來，使用Undo Last commit就可以回到之前的狀態

點選VSCode Sourse Control(Ctrl+Shift+G)(第3個圖示)框右上角三個點，點選Commit，再點選Undo last Commit，可以修改commit內容或是檔案內容，也等於取消commit，並回到上一次commit之前。使用指令:git log 看commit紀錄，會發現上一個commit已經不見了。如果要在git command line使用，可以用指令:git reset --soft HEAD~1，能往前退一個commit，並保留檔案修改的部分，只取消commit，不影響變更後的內容。指令:git reset --soft HEAD~2，能往前退二個commit，但是只能退到剩一個commit。

指令:git pull，把預設的upstream的新進度抓下來(現在的upstream是origin，也就是剛剛見的github repo)

指令:git log
可以看到git log這邊就已經是有chapter 2，就是A做的修改的部分，剛剛A是在master上面做修改，這邊切到master分支才抓下來，A修改後上傳到github，B使用git pull把A的新版本抓到B電腦的master分支，表示主分支master會是最新的版本，才能做rebaase。簡單來說就是先把master pull一次，再回到new-feature branch上做rebase。

指令:git switch new-feature

  • git rebase --continue，解完衝突之後繼續rebase(有時候可能要解複數次衝突)

指令:git push
不能push，因為push在一般的情況下，只允許對一個repository去做新的commits。現在電腦端的git log跟github上的commits紀錄是不相容的，也就是說push上去會改變歷史紀錄，一般push只允許新增新的commits在後面，如果要強制push，可以使用指令:git push -f，強制把現在電腦上的紀錄push到remote。

注意事項:
指令:git push -f，非常不建議在主分支使用，只能在自己新開出來的分支上做，因為一旦改變了過去的歷史，每一個在branch上面開發的所有人都會受到影響，會讓整個repository混亂，所以不要在大家共用的主要分枝上做這件事情，只能在自己新開的分支上面做!!!

(已經將電腦端的commits，push到github上了)
成功push上去之後，再次進入github new-feature branch，點選Compare & pull request(綠色按鈕)或是Contribute項下的Open pull requesst。這次就可以Able to merge.，再點選Create pull request(綠色按鈕)，再點選Merge pull requst(綠色按鈕)項，有三種不同方法，通常是使用第一種Create a merge commit，並點選Confirm merge(綠色按鈕)。

小整理:

1.在github上用pull request

指令:git switch master
指令:git pull
指令:git switch new-feature
指令:git rebase master
指令:git push (conflicts)
指令:git push -f (在new-feature)
在github pull request

2.在自己的電腦merge之後再push

指令:git switch master
指令:git merge new-feature2
指令:git log(merge少了A的更新)
指令:git push (conflicts)
不能用指令:git push -f (在master)
指令:git pull
指令:git push

總結:
較不推薦使用第二種方式，因為用git pull處理conflict，沒有git rebase處理conflict來得好，git pull無法確保merge前就把所有衝突解決，雖然merge前可以先切回master再pull一次，是不是就不會產生conflict，但要是merge之後，還沒push，剛好有人在這時候更新主要分支，之後又會有衝突要處理，這時候電腦端new-feature2已經merge回master，自己的分支已經不見了，conflict的處理就會比較麻煩。如果使用pull request，github會確保branches一定要在最新進度才能merge，使用pull request可以確保先在自己的分支解完衝突才執行merge，只要多rebase一次就可以解決有conflict的問題。
